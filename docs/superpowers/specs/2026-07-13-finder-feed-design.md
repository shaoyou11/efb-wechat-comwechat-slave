# 微信视频号分享消息兼容设计

## 目标

兼容 ComWeChat 返回的 `share` 消息中 `appmsg/type=51`、包含
`finderFeed` 的微信视频号分享。Telegram 不再只显示“当前微信版本不支持展示该内容”，
而是收到可播放视频及对应作者、文案。

## 范围

- 仅修改 `efb_wechat_comwechat_slave` 的分享消息解析。
- 不修改 ComWeChat、Telegram Master、Bot API、watchdog、登录逻辑和文件大小配置。
- 不改变普通公众号链接、小程序分享及其他现有消息类型的行为。

## 数据流

1. `MsgProcess` 继续将 `share` 消息交给现有分享包装器。
2. 分享包装器识别 `appmsg/type=51` 且存在 `finderFeed`。
3. 从 `finderFeed` 提取作者、文案、视频时长、封面和第一条视频地址。
4. 视频地址存在时立即下载到临时文件，构造 EFB `Video` 消息。
5. 视频消息正文包含作者和原始文案。
6. 视频下载或解析失败时，发送包含作者、文案和封面的降级消息；不得回退到微信升级提示。

## 错误处理

- 缺少 `finderFeed`：保持现有分享消息处理逻辑。
- 缺少视频地址：使用封面和文字降级。
- 视频 CDN 超时、拒绝访问或内容为空：关闭临时资源并使用封面和文字降级。
- XML 字段缺失：允许作者、文案、封面分别为空，不因单个字段缺失丢弃整条消息。
- 不持久保存微信 CDN 临时视频；发送结束后沿用 EFB 临时文件生命周期。

## 测试

- 使用本次真实消息结构构造脱敏测试样本。
- 验证 `type=51 + finderFeed` 可提取作者、文案、视频地址、封面和时长。
- 验证视频下载成功时生成 `Video` 消息。
- 验证下载失败和缺少视频地址时生成降级消息。
- 验证普通 `type=5` 分享仍走原有逻辑。
- 部署前运行仓库完整测试和 Python 编译检查。

## 部署与回退

1. 备份 NAS 当前 Compose、EFB 配置及当前镜像标识。
2. 构建并推送 `ghcr.io/shaoyou11/efb:latest`，同时保留固定回退标签。
3. 只重建 `efb2026-efb-1`。
4. 用新发的视频号消息核对 EFB 日志和 Telegram 展示结果。
5. 如验证失败，恢复原镜像标识并只重建 EFB 容器。
